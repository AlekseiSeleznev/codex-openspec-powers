## Context

Codex OpenSpec Powers combines OpenSpec workflow discipline with Codex prompts,
optional skills, Superpowers recommendations, and an opt-in external review
loop. The target user wants a reusable overlay that can be installed on top of
an existing OpenSpec project.

The local project has been initialized as a separate repository with
`openspec init --tools codex`. This produced a Codex-compatible OpenSpec base
under `.codex/skills` and `openspec/config.yaml`. The change should build on
that Codex/OpenSpec base and keep the template focused on its own workflow
contract.

Primary stakeholders:

- A Codex user who wants repeatable OpenSpec workflows in existing projects.
- Existing OpenSpec projects that may already have project-specific schemas,
  specs, active changes, and Codex or agent instructions.

## Goals / Non-Goals

**Goals:**

- Produce a Codex-native version of the Codex OpenSpec Powers template.
- Make the template usable as an overlay for existing OpenSpec projects.
- Adapt useful reusable skills into `.codex/skills`.
- Provide Codex prompt files for the `opsx` workflow.
- Preserve OpenSpec artifact discipline: proposal, specs, design, tasks,
  implementation, verification, archive.
- Document conflict points so installation does not silently overwrite
  project-owned OpenSpec content.
- Provide a repair prompt that can be run after `openspec init --tools codex`
  or OpenSpec upgrades to verify and restore the overlay safely.
- Define a manifest-driven ownership model so repair can distinguish
  OpenSpec-generated files and template-owned additions.
- Provide an opt-in Claude OpenSpec Review loop for external artifact review at
  proposal, apply-readiness, and archive-readiness gates.

**Non-Goals:**

- Implement an automated installer script in this change unless later tasks
  prove it is necessary.
- Replace a target project's existing OpenSpec schema without explicit user
  action.
- Build application features unrelated to the template workflow.

## Decisions

### Decision: Treat OpenSpec as the stable core and Codex as the overlay

The target template will separate durable OpenSpec assets from Codex-specific
assets:

```text
OpenSpec core
  openspec/config.yaml
  openspec/schemas/*              optional, merge-reviewed

Codex overlay
  .codex/prompts/opsx-*.md
  .codex/skills/*
  .codex/codex-openspec-powers/manifest.yaml
  AGENTS.md
  README.md
  INSTALL_CODEX_TEMPLATE.md
```

Rationale: existing projects may already have OpenSpec state. Treating Codex as
an overlay avoids turning installation into a destructive project rewrite.

Alternative considered: keep all guidance in prose only. That is insufficient
because prompts, skills, and manifest entries are easier to verify and repair
than implicit documentation.

### Decision: Port command behavior into `.codex/prompts`

The target prompt layer will use Codex command naming such as `/opsx:new`,
`/opsx:propose`, `/opsx:continue`, `/opsx:explore`, `/opsx:apply`,
`/opsx:verify`, `/opsx:sync`, `/opsx:archive`, `/opsx:ff`,
`/opsx:bulk-archive`, `/opsx:check-overlay`, and `/opsx:claude-review`.

Command intent:

- `/opsx:new`: scaffold a new OpenSpec change and show the first artifact
  instructions.
- `/opsx:propose`: create the full proposal/spec/design/tasks artifact set in
  one guided pass when the user wants fast-forward artifact creation.
- `/opsx:continue`: create the next ready artifact for an existing change.
- `/opsx:explore`: think and inspect without implementation.
- `/opsx:apply`: implement tasks from a ready change.
- `/opsx:verify`: verify implementation against proposal, specs, design, and
  tasks.
- `/opsx:sync`: sync accepted spec deltas without archiving.
- `/opsx:archive`: archive a completed change.
- `/opsx:ff`: fast-forward OpenSpec artifact creation for an active change.
- `/opsx:bulk-archive`: archive multiple completed changes.
- `/opsx:check-overlay`: inspect the overlay and propose safe repair if needed.
- `/opsx:claude-review`: control, inspect, and manually run the optional
  Claude OpenSpec Review loop.

Prompt content should use Codex-compatible language, Codex slash-command names,
and the existing Codex-oriented OpenSpec prompts available in this environment.

Alternative considered: rely only on `.codex/skills` generated by OpenSpec.
That is insufficient because slash-command prompts are the user-facing workflow
entry points the template is meant to provide.

### Decision: Adapt skills selectively

The target skill set will include only skills that support the Codex/OpenSpec
workflow. These engineering skills are bundled as a library and lazy-loaded
only after user consent:

- OpenSpec lifecycle skills.
- ADR authoring.
- C4 diagram guidance.
- Gherkin requirement authoring.
- OpenSpec git discipline.
- Rigorous design review with project documents (`grill-with-docs`).

Skills must be rewritten or reviewed for Codex compatibility before inclusion.
Optional skills are installed as a library, not treated as always-loaded
context.

### Decision: Optional steps use recommendation prompts

When a workflow reaches an optional step such as `grill-with-docs`, C4 diagram
creation, ADR extraction, or Gherkin refinement, Codex should not silently run
the step. It should announce a short recommendation, explain the trigger, and
ask for consent before loading the optional skill. If the Codex environment
supports an interactive user-question control, the workflow should use it;
otherwise it should render the same choice as a compact Markdown block.

Accepted ordinary skills, Superpowers steps, and review tools are substeps of
the active OpenSpec workflow. After a recommended substep finishes, Codex must
return to the active workflow and still emit the final `**== УРОКИ ==**` and
`**== СЛЕДУЮЩИЙ ШАГ ==**` blocks for the user-facing response.

Recommended Markdown fallback shape for ordinary skills:

```markdown
**== РЕКОМЕНДАЦИЯ ==**

Вызвать `grill-with-docs`

Change затрагивает prompts, skills и install workflow, поэтому стоит проверить
proposal по документам.

Выполнить?

**Да / Нет**
```

Superpowers use a separate wording:

```markdown
**== РЕКОМЕНДАЦИЯ ==**

Использовать `superpowers:writing-plans`

Tasks затрагивают prompts, skills и installation docs, поэтому перед apply стоит
разложить реализацию в более точный execution plan.

Использовать?

**Да / Нет**
```

The recommendation should be proportional:

```text
Low complexity     -> "I think we can skip this."
Medium complexity  -> "Optional; useful if you want a second pass."
High complexity    -> "Recommended because <specific reason>."
Critical risk      -> "Strongly recommended; skipping is allowed if you confirm."
```

Rationale: this keeps context loading explicit and gives the user control while
still letting Codex exercise engineering judgment.

Alternative considered: preserve `.agents/skills` as-is. That keeps useful
content, but it fails the Codex-native goal and forces future users to discover
which parts are actually safe for Codex.

### Decision: Integrate Superpowers as optional methodology accelerators

Superpowers will be recommended only when they fit the current OpenSpec step.
They do not replace OpenSpec artifacts.

Supported recommendation targets:

- `superpowers:brainstorming`
- `superpowers:writing-plans`
- `superpowers:test-driven-development`
- `superpowers:systematic-debugging`
- `superpowers:verification-before-completion`
- `superpowers:requesting-code-review`
- `superpowers:finishing-a-development-branch`

Superpowers recommendations use `Использовать` and `Использовать?`, while
ordinary skills use `Вызвать` and `Выполнить?`.

### Decision: Add opt-in Claude OpenSpec Review

The overlay will support an optional external Claude review loop. It is OFF by
default and can be controlled through:

- `/opsx:claude-review on`
- `/opsx:claude-review off`
- `/opsx:claude-review status`
- `/opsx:claude-review run <change> <stage>`

Claude review stages:

- `propose`: after `/opsx:propose`, budget `$0.50`.
- `apply-readiness`: before `/opsx:apply`, budget `$1.50`.
- `archive-readiness`: after `/opsx:verify` and before `/opsx:archive`, budget
  `$1.00`.

Default Claude review config lives in
`.codex/codex-openspec-powers/claude-review.yaml`:

```yaml
claudeOpenSpecReview:
  enabled: false
  model: default
  effort: default
  maxBudgetUsd:
    propose: 0.50
    applyReadiness: 1.50
    archiveReadiness: 1.00
  # Optional per-stage model/effort overrides use canonical camelCase YAML keys.
  # Omit a stage block when no override is needed.
  # stageOptions:
  #   propose:
  #     model: default
  #     effort: default
  #   applyReadiness:
  #     model: default
  #     effort: default
  #   archiveReadiness:
  #     model: default
  #     effort: default
  autoApplySafeDocFixes: false
  blockOnCriticalHigh: true
  saveReviews: true
  preCallLimits:
    propose:
      maxInputBytes: 200000
      maxFiles: 30
    applyReadiness:
      maxInputBytes: 400000
      maxFiles: 50
    archiveReadiness:
      maxInputBytes: 300000
      maxFiles: 50
```

Because `openspec/config.yaml` may not support arbitrary top-level keys, this
configuration belongs in the overlay-owned Claude review config file, not
directly as an OpenSpec config extension.

`model` and `effort` are pass-through settings for the local Claude CLI
environment. The value `default` is a sentinel meaning "omit the override and
inherit Claude CLI defaults"; Codex must not pass literal `default` as a model
or effort value. Before invoking Claude, Codex displays the resolved stage
budget, model, effort, and pre-call limits in the `**== РЕВЬЮ CLAUDE ==**`
block.

Claude receives only relevant change files by default. Codex enforces this by
creating a temporary isolated review bundle containing only whitelisted files
and invoking `claude -p` from that bundle directory, never from the repository
root.

- `openspec/changes/<change>/proposal.md`
- `openspec/changes/<change>/design.md`
- `openspec/changes/<change>/tasks.md`
- `openspec/changes/<change>/README.md`
- `openspec/changes/<change>/specs/*/spec.md`
- `openspec/config.yaml`

Claude never edits the repository directly. Codex may apply only safe doc-only
recommendations and records any applied summary. With the default
`autoApplySafeDocFixes: false`, Codex shows safe fixes and asks before applying
them. Critical/High findings block `/opsx:apply` and `/opsx:archive` until a
stable finding ID is recorded in the canonical applied log or manual resolution
file. Re-running Claude is not enough to clear a prior blocker if the finding is
not mentioned in the later output.

Claude review is a preliminary substep. Even when Claude review runs, skips, or
blocks the workflow, Codex still ends the overall response with
`**== УРОКИ ==**` and `**== СЛЕДУЮЩИЙ ШАГ ==**`.

### Decision: Emit Russian UX markers in v1

The overlay will use exact Russian UX markers in v1:

- `**== РЕКОМЕНДАЦИЯ ==**`
- `**== УРОКИ ==**`
- `**== СЛЕДУЮЩИЙ ШАГ ==**`
- `Выполнить?`
- `Использовать?`
- `**Да / Нет**`

Localization is out of scope for this change. A future change can parameterize
or translate the UX markers after the Codex overlay behavior is stable.

### Decision: Installation guidance is part of the product

The template will include an `INSTALL_CODEX_TEMPLATE.md` file that describes
how to apply the overlay to an existing project. It should classify paths as:

- Safe to add when absent.
- Merge-review required.
- Do not overwrite without explicit backup and user decision.

Rationale: the user's stated goal is to install this over existing OpenSpec
projects, so installation behavior is a first-class requirement.

### Decision: Add an overlay healthcheck/repair prompt

The template will include a Codex prompt named `/opsx:check-overlay` that
validates the current OpenSpec/Codex installation against the codex-openspec-powers
overlay contract.

The prompt should:

1. Inspect current `.codex/`, `AGENTS.md`, and `openspec/config.yaml`.
2. Load the overlay manifest.
3. Detect missing overlay-owned files and missing required prompt fragments.
4. Detect manifest, marker, snippet, and documentation drift.
5. Compare planned repairs against existing content.
6. Report the repair plan before writing.
7. Ask for confirmation.
8. Apply only overlay-owned additions or explicitly approved merges.
9. Run validation checks after repair.

The repair prompt should preserve OpenSpec-generated updates wherever possible.
It is not a rollback tool and should not blindly restore generated files.

### Decision: Make overlay repair manifest-driven

`.codex/codex-openspec-powers/manifest.yaml`, that classifies:

- Template-owned paths.
- Required markers.
- Required snippets.
- OpenSpec-generated paths.
- Merge-review paths.
- Never-overwrite paths.

Healthcheck and repair use this manifest as their authority. They must not infer
ownership from filenames alone.

The YAML stage keys are intentionally camelCase: `propose`,
`applyReadiness`, and `archiveReadiness`. CLI stage names remain kebab-case
(`propose`, `apply-readiness`, `archive-readiness`) and are mapped to these
configuration keys by the overlay. Kebab-case stage keys in
`claude-review.yaml` are treated as ignored unknown keys and should produce a
warning rather than silently acting as overrides.

Paths are repository-relative. Glob patterns are allowed only where explicitly
shown in the manifest. Specific file matches in `templateOwnedPaths` take
precedence over containing-directory matches in `mergeReviewPaths`, so repair
can restore template-owned prompt files without treating user-added files in the
same directory as template-owned. Required snippets are minimal marker checks,
not full file replacements. Required prompt paths are presence checks; they
allow the healthcheck to detect an entirely missing prompt before checking
prompt markers.

`openspec-git-discipline` is an ordinary optional skill that can be
recommended from `/opsx:apply` and `/opsx:archive` before commit, branch, or
archive-related git operations. Those prompts therefore participate in the
ordinary-skill consent snippet check even when no git operation is ultimately
performed.

Template-owned repair sources are the overlay's bundled template content:
the prompt, config, manifest, documentation, and skill files shipped by this
template. If that bundled source is unavailable in the current installation,
`/opsx:check-overlay` reports the missing template-owned file and instructs the
user to re-apply the template rather than inventing content.

Conflict definition: a generated or project-owned file conflicts with the
overlay only when it lacks, removes, omits, alters, or contradicts a
manifest-required marker, required snippet, canonical command syntax, or
consent wording. Other generated updates are preserved. Paths not matched by
any manifest section are left untouched and may be reported as informational,
not as broken overlay content.

Recommended user interaction:

```text
/opsx:check-overlay
Проверь свежую установку OpenSpec/Codex и покажи, какие части
Codex OpenSpec Powers overlay отсутствуют или были перетёрты.
```

## Risks / Trade-offs

- [Risk] Existing projects may already have `.codex` or `AGENTS.md` content.
  -> Mitigation: document these as merge-review paths and avoid installer
  instructions that blindly replace them.
- [Risk] Optional project-specific schema support may conflict with a target
  project's existing OpenSpec schema.
  -> Mitigation: treat schema installation as optional and explicitly
  user-controlled.
- [Risk] Optional skills may bloat Codex context if prompts load them
  unconditionally.
  -> Mitigation: prompts should load C4, ADR, Gherkin, and grill-with-docs only
  when the change complexity or task type justifies the context cost.
- [Risk] Frequent optional prompts may annoy users.
  -> Mitigation: prompt only at meaningful decision points and include a clear
  default recommendation so the user can answer quickly.
- [Risk] The overlay repair prompt may guess ownership incorrectly.
  -> Mitigation: use a manifest and stop for merge review when ownership is
  missing or ambiguous.
- [Risk] A future installer script could add complexity before the manual
  workflow is stable.
  -> Mitigation: use documented manual installation in v1; consider an
  installer script in a later change after the overlay contract stabilizes.

## Implementation Plan

1. Inventory template files and classify each as keep, adapt, or drop.
2. Inventory generated local Codex/OpenSpec files and identify the target prompt
   and skill names.
3. Add the overlay manifest.
4. Add or adapt Codex prompts for the `opsx` lifecycle.
5. Adapt selected skills into `.codex/skills`.
6. Add the optional Claude review prompt/config and review storage convention.
7. Populate supported `openspec/config.yaml` `context` and `rules` with overlay
   methodology guidance without adding nonstandard top-level keys.
8. Add `README.md`, `AGENTS.md`, and `INSTALL_CODEX_TEMPLATE.md` describing the
   Codex-native workflow and overlay installation process.
9. Add the overlay healthcheck/repair prompt and document when to run it.
10. Validate OpenSpec commands and proposal/apply lifecycle against the target
   structure.

Rollback is straightforward while this repository is a template: revert the
change or remove the affected `.codex`, documentation, and optional schema
assets before publishing or applying it elsewhere.
