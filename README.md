Codex OpenSpec Powers добавляет к OpenSpec слой для работы в Codex: команды
`/opsx:*`, локальные навыки, правила безопасной установки, манифест надстройки,
проверку состояния, рекомендации Superpowers и опциональную проверку Claude
OpenSpec Review.

OpenSpec остается источником истины. Предложения, спецификации, проектные
решения, списки задач, реализация, проверка, синхронизация и архивация проходят
через артефакты OpenSpec.

## Главное

- Команды `/opsx:*` для полного жизненного цикла OpenSpec в Codex.
- Проверка надстройки по манифесту с сохранением проектных файлов.
- Дополнительные навыки подключаются только после рекомендации и согласия.
- Если на шаге есть несколько необязательных действий, Codex показывает один
  общий блок рекомендации и принимает выбор `1`, `1,2`, `+` или `-`, а
  следующий шаг учитывает выбранные или пропущенные действия.
- Интеграция с плагином Superpowers как с необязательным набором
  методологических ускорителей.
- Проверочные этапы Claude OpenSpec Review с изолированным набором проверяемых
  файлов.
- Завершенные пользовательские шаги OpenSpec заканчиваются блоками
  `УРОКИ` и `СЛЕДУЮЩИЙ ШАГ`.
- Опубликованные спецификации OpenSpec, описывающие поведение самой
  надстройки.

## Команды

Файлы команд находятся в `.codex/prompts`.

| Команда | Назначение |
| --- | --- |
| `/opsx:init` | Инициализировать новый проект из Codex: запустить `opsx init`, восстановить надстройку и проверить состояние. |
| `/opsx:new` | Создать изменение OpenSpec. |
| `/opsx:propose` | Подготовить предложение, спецификации, проектное решение и задачи. |
| `/opsx:continue` | Продолжить существующее изменение. |
| `/opsx:explore` | Исследовать задачу без перехода к реализации. |
| `/opsx:apply` | Реализовать изменение по артефактам OpenSpec. |
| `/opsx:verify` | Проверить реализацию. |
| `/opsx:sync` | Синхронизировать изменения спецификаций. |
| `/opsx:archive` | Архивировать завершенное изменение. |
| `/opsx:ff` | Быстро подготовить артефакты OpenSpec. |
| `/opsx:bulk-archive` | Архивировать несколько изменений. |
| `/opsx:check-overlay` | Проверить надстройку. |
| `/opsx:claude-review` | Управлять проверкой Claude: `on`, `off`, `status`, `run`. |

## Навыки

В `.codex/skills` собраны навыки, которые расширяют рабочий процесс OpenSpec в
Codex. Они помогают провести изменение от идеи до архивации, уточнить
требования, зафиксировать архитектурные решения и проверить проектные
артефакты перед реализацией.

### Навыки жизненного цикла OpenSpec

Эти навыки поддерживают основные шаги жизненного цикла OpenSpec:

- `openspec-new-change` — создание нового изменения.
- `openspec-propose` — подготовка предложения, спецификаций, проектного решения и списка задач за один проход.
- `openspec-continue-change` — продолжение существующего изменения и создание следующего артефакта.
- `openspec-explore` — исследование задачи без перехода к реализации.
- `openspec-apply-change` — реализация задач, описанных в изменении OpenSpec.
- `openspec-verify-change` — проверка реализации по предложению, спецификациям, проектному решению и списку задач.
- `openspec-sync-specs` — перенос изменений из временных спецификаций в основные спецификации.
- `openspec-archive-change` — архивация завершенного изменения.
- `openspec-bulk-archive-change` — архивация нескольких завершенных изменений.
- `openspec-ff-change` — быстрая подготовка всех артефактов OpenSpec до готовности к реализации.
- `openspec-onboard` — пошаговое знакомство с рабочим процессом OpenSpec на реальной задаче.

### Дополнительные инженерные навыки

Эти навыки не заменяют OpenSpec, а дополняют его там, где нужны более точные
проектные артефакты или дополнительная инженерная дисциплина:

- `architectural-decision-records` — фиксирует одно архитектурно значимое решение за раз: причины, компромиссы, альтернативы, последствия, статус и историю изменения решения.
- `c4-diagrams` — описывает архитектурные границы, зоны ответственности, участников, зависимости и потоки данных, а затем строит только тот уровень C4-диаграммы, который нужен для текущего вопроса.
- `gherkin-authoring` — превращает ожидаемое поведение в конкретные примеры на языке Gherkin с понятным исходным состоянием, одним значимым событием и наблюдаемым результатом.
- `grill-with-docs` — выполняет контекстную стресс-проверку предложения, проектного решения или плана по артефактам OpenSpec, записям архитектурных решений, проектной документации и релевантному коду, затем задает точечные вопросы по главным рискам и предлагает рекомендуемые ответы.
- `openspec-git-discipline` — следит за Git-дисциплиной вокруг жизненного цикла OpenSpec: состояние предложения должно попасть в `main` до реализации, архивация выполняется из слитого `main`, а коммиты и слияния создаются только по явному запросу.

Дополнительные навыки подключаются только после рекомендации и явного согласия
пользователя. Superpowers в этот набор не входят: они не поставляются вместе с
репозиторием и требуют отдельного плагина Superpowers в окружении Codex.

Когда на одном этапе подходят несколько необязательных действий, команды
`/opsx:*` показывают один общий блок `РЕКОМЕНДАЦИЯ` с нумерованными пунктами.
Пользователь может выбрать один пункт, последовательность пунктов, все пункты
или ничего: `1`, `1,2`, `+`, `-`. После выбранных действий Codex возвращается к
активному шагу OpenSpec и предлагает следующий шаг уже с учетом результата.

## Claude OpenSpec Review

Claude OpenSpec Review выключен по умолчанию:

```text
/opsx:claude-review on
/opsx:claude-review off
/opsx:claude-review status
/opsx:claude-review run <change> <stage>
```

Этапы: `propose`, `apply-readiness`, `archive-readiness`.
Конфигурация: `.codex/codex-openspec-powers/claude-review.yaml`.
Результаты: `openspec/changes/<change>/reviews/`.

Claude получает набор проверяемых файлов, по умолчанию ограниченный
`proposal.md`, `design.md`, `tasks.md`, `README.md`, `specs/*/spec.md` для
выбранного изменения и `openspec/config.yaml` из изолированного временного
пакета. Проверка может расходовать API-кредиты и ограничивается настроенными бюджетами
по этапам. Неразрешенные замечания уровня Critical/High блокируют
`/opsx:apply` и `/opsx:archive`, пока запись о примененном исправлении или
ручное решение не зафиксирует идентификатор замечания.

## Установка

Рекомендуемый автоматический способ:

```bash
bin/opsx install-auto-repair
```

После этого для нового проекта достаточно выполнить:

```bash
opsx init /path/to/project
```

или из корня проекта:

```bash
opsx init
```

Команда сама запускает `openspec init --tools codex`, накладывает надстройку и
проверяет, что команды `/opsx` не перезаписаны. Установленная обёртка также
перехватывает обычный вызов:

```bash
openspec init --tools codex
```

и автоматически выполняет восстановление надстройки после успешного `init`.

Если вы начинаете проект прямо из Codex, можно использовать команду:

```text
/opsx:init
```

Она вызывает `opsx init`, затем `opsx doctor` и предлагает перейти к
`/opsx:explore` для первого описания идеи проекта.

Ручной состав надстройки:

```text
.codex/codex-openspec-powers/
.codex/prompts/opsx-*.md
.codex/skills/architectural-decision-records/
.codex/skills/c4-diagrams/
.codex/skills/gherkin-authoring/
.codex/skills/grill-with-docs/
.codex/skills/openspec-git-discipline/
INSTALL_CODEX_TEMPLATE.md
```

Если `.codex`, `AGENTS.md`, `openspec/config.yaml` или `openspec/schemas` уже
существуют, выполняйте проверку слияния. Не заменяйте без явного решения
пользователя `openspec/changes/`, `openspec/specs/`, исходный код, тесты,
файлы пакетов, сборки и конфигурации, а также проектную документацию.

Для ручной проверки после установки выполните:

```text
/opsx:check-overlay
```

Если `openspec init --tools codex` или обновление OpenSpec перезапишет команды
`/opsx`, автоматическое восстановление вернёт файлы, принадлежащие шаблону, из
встроенного набора исходных файлов
`.codex/codex-openspec-powers/template/`. `/opsx:check-overlay` остаётся ручной
проверкой и безопасным способом увидеть план восстановления.

Подробная инструкция находится в
[INSTALL_CODEX_TEMPLATE.md](INSTALL_CODEX_TEMPLATE.md).

## Структура

```text
bin/                              командная утилита `opsx` и обёртка для автоматического восстановления
.codex/
  prompts/                         команды Codex
  skills/                          навыки жизненного цикла OpenSpec и дополнительные навыки
  codex-openspec-powers/           манифест, конфигурация Claude и встроенный набор исходных файлов для восстановления
openspec/
  config.yaml                       контекст и правила OpenSpec для надстройки
  specs/                           опубликованные требования надстройки
  changes/archive/                 архивированное изменение реализации
README.md                          русская версия и главная страница GitHub
README.en.md                       английская версия
```

## Проверка

```bash
openspec validate --all --strict
openspec list --specs
```
